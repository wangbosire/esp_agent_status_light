# AGENT.md

## 1. 目的

本仓库要求所有 AI 编码代理在分析、设计、修改、修复、重构和扩展代码时，必须严格遵循项目技术方案，不允许脱离技术方案自行发挥。

本项目实现的第一事实来源是：

- [docs/TECHNICAL_DESIGN.md](docs/TECHNICAL_DESIGN.md)

辅助参考文档包括：

- [docs/ARCHITECTURE.md](docs/ARCHITECTURE.md)
- [README.md](README.md)
- [docs/USER_MANUAL.md](docs/USER_MANUAL.md)

如果现有代码、历史实现习惯、局部注释、口头默认约定与 `docs/TECHNICAL_DESIGN.md` 不一致，在用户没有明确要求变更方案的前提下，AI 必须以 `docs/TECHNICAL_DESIGN.md` 为最高准则。

## 2. 最高规则

AI 的职责是实现技术方案，不是重写技术方案。

AI 不得因为“更方便”“更常见”“更符合个人经验”“实现起来更快”而绕开、弱化或替换技术方案中已经明确写出的架构、边界、协议、语义、优先级、兜底策略和模块职责。

一句话要求：

- 先读方案，再实现。
- 按方案实现，不按想象实现。
- 方案没有写清楚时，先升级确认，不允许擅自补规则。

## 3. 开工前强制流程

在做任何代码修改前，AI 必须先完成以下动作：

1. 阅读 `docs/TECHNICAL_DESIGN.md` 中与当前任务直接相关的章节。
2. 确认本次修改涉及的模块边界、port、adapter、命令语义或状态规则。
3. 判断该需求是否已经被技术方案覆盖。
4. 仅在技术方案允许的边界内实施修改。

如果需求没有被方案明确覆盖，或者方案存在歧义，AI 必须：

1. 暂停“自行设计”行为。
2. 明确指出缺失或冲突的方案位置。
3. 请求用户确认，或先更新技术方案，再继续实现。

## 4. 规则优先级

实现时必须按以下优先级决策：

1. `docs/TECHNICAL_DESIGN.md`
2. 当前任务中用户明确提出的要求
3. 已有代码中与技术方案一致的稳定实现模式
4. 其它补充文档，如 `ARCHITECTURE.md`、`README.md`、`USER_MANUAL.md`

注意：

- “代码现在就是这么写的” 不是偏离技术方案的理由。
- “之前某次改动这么做过” 也不是偏离技术方案的理由。

## 5. 架构硬约束

AI 必须严格保留技术方案定义的 Ports and Adapters 架构，不允许把边界打穿。

具体要求如下：

- 核心域只能围绕稳定概念实现：`AgentEvent`、`AgentCapability`、`Mode`、`SourceState`、TTL、优先级和 effective mode。
- 所有外部差异必须落在 adapter 中，包括但不限于：
  - Agent Hook stdin 解析
  - Hook 安装与卸载
  - IPC 传输
  - BLE / 设备访问
  - runtime 存储
  - 日志实现
  - 平台差异
- CLI 层只负责参数解析、adapter 装配和命令分发，不得堆积核心业务规则。
- daemon、router、model、ports 等核心模块不得泄漏第三方宿主工具的私有 JSON schema。
- Hook 原始 stdin 的解析必须发生在命令侧和 `SourceAdapter` 内，不得挪入 daemon 或 router。
- IPC 上传输的必须是归一化后的稳定数据，不得传原始第三方 Hook JSON。
- 新增 Agent 时，应优先新增 `adapters/source/<agent>.rs`、`adapters/install/<agent>.rs` 并注册到 registry，而不是修改历史核心逻辑。

## 6. 明确禁止的行为

在用户没有明确要求修改技术方案的前提下，AI 不得做以下事情：

- 绕过 ports / adapters 边界，直接把外部实现细节写进核心层。
- 把 source 特定逻辑塞进 `router`、`daemon`、`model` 或核心 port。
- 改写 CLI 命令语义，使其与方案不一致。
- 擅自改变 mode 优先级、TTL、off 语义、effective mode 计算规则或 fallback 策略。
- 使用猜测出来的第三方事件行为替代技术方案中的官方 rule table。
- 在技术方案明确说明“某字段只能作为兼容兜底”时，把它提升为主判断依据。
- 把 TCP loopback 从“调试降级方案”改成默认 IPC。
- 把 `install/uninstall` 的 target 改成写死枚举，如果方案要求走 registry。
- 因为实现方便就修改模块职责边界。

## 7. 新增能力时的实现要求

如果任务涉及新增 Agent、平台行为、IPC 方式、存储后端或日志后端，AI 必须优先沿着技术方案已经预留的扩展缝实现：

- 新 Agent：新增 source adapter、install adapter，并注册到对应 registry
- 新平台行为：扩展 `PlatformAdapter`
- 新 IPC：实现既有 `IpcTransport` / `IpcServer` port
- 新 runtime/log backend：实现对应 port

除非技术方案明确要求修改核心模块，否则 AI 不应为新增能力改动历史核心逻辑。

## 8. 修 Bug 时的要求

修复问题时，AI 必须先判断“这是相对于技术方案的 bug，还是只是当前实现和方案不一致的暴露”。

修 Bug 前必须先问自己：

1. 当前行为是否真的违反了 `docs/TECHNICAL_DESIGN.md`？
2. 这个修复应该落在核心层，还是应该落在 adapter 边界？
3. 修复是否会改变已文档化的 CLI、Hook、IPC、状态路由或安装行为？
4. 修复是否会把本应隔离在 adapter 的逻辑泄漏到核心？

如果一个 bug 的正确修法会超出当前技术方案，AI 必须明确指出，而不是静默“顺手扩展设计”。

## 9. 注释与文档要求

当 AI 修改代码时，应优先补充“为什么这样实现”的注释，而不是重复语法表面含义。

凡是新增逻辑、修复 bug、改变状态语义、增加兜底策略、处理并发/锁/超时/缓存/恢复路径，AI 必须在同一次代码修改中同步补充关键注释，说明设计意图和边界条件，不得等用户再次提醒。

优先解释以下内容：

- 这一段实现对应技术方案中的哪条规则
- 为什么该逻辑必须放在 adapter，而不能放进核心层
- 为什么这里采用某个 fallback / 兜底策略
- 为什么某个分支是在保护技术方案中的状态语义、兼容性或边界约束
- 为什么某个并发锁、超时、缓存刷新或 stale runtime 清理策略是安全的
- 如果某条路径是为现场问题或历史兼容性保留，必须说明触发场景和不这么做的风险

不鼓励无意义注释，例如“把值赋给变量”这类重复代码字面含义的描述。

## 10. 验证要求

实现完成后，AI 应主动检查结果是否仍然符合技术方案。

至少应验证以下内容：

- 模块放置位置是否符合架构分层
- 新增行为是否符合文档中的命令语义和 Hook 语义
- registry 注册是否完整
- 测试覆盖的是否是技术方案规则，而不只是表面输出
- 是否有第三方宿主的原始 schema 泄漏到 daemon/router/core

如果本次修改涉及状态机、安装器、IPC、source adapter 或 daemon 行为，AI 应优先补或更新对应测试。

## 11. 汇报要求

AI 在总结改动时，应明确说明：

1. 本次实现依据了技术方案的哪些章节
2. 本次改动是否完全落在现有技术方案内
3. 是否存在技术方案不明确、需要用户决策的地方

不允许把明显超出方案的行为包装成“顺手优化”或“工程细节调整”。

## 12. 冲突升级规则

如果用户提出的需求与 `docs/TECHNICAL_DESIGN.md` 冲突，AI 不得直接无提示地实现冲突行为。

AI 必须：

1. 明确指出冲突点
2. 说明冲突涉及的技术方案范围
3. 询问用户是要：
   - 更新技术方案后再实现
   - 明确接受一次设计例外

在用户未明确确认前，AI 不得私自偏离方案落地代码。

## 13. 本项目的关键执行提醒

针对本仓库，AI 需要特别牢记以下几条：

- `command` 层只做装配与分发，不承载核心业务判断
- `router` 只处理统一状态、TTL、优先级和 effective mode
- `daemon` 只处理归一化后的请求，不解析各家 Hook 原始 stdin
- `SourceAdapter` 负责各家 Hook stdin 的 schema 适配与能力归一
- `HookInstallAdapter` 负责各家配置文件 schema 的安装与卸载
- 新接入 Agent 时，原则上只新增 adapter 和注册表，不回改历史核心
- 非官方字段只能用于兼容兜底，不能成为主 rule table 的事实来源
- Hook 失败不能阻塞 Agent 主流程，这条约束不能被任何“严格模式之外”的实现破坏

## 14. 极简版规则

先读技术方案。

严格按技术方案实现。

不要脑补设计，不要偷改边界，不要擅自发明规则。

方案缺失、冲突或不清楚时，先升级确认，再继续实现。
